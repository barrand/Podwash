# PodWash matching & interval spec

> **Provenance:** Extracted verbatim from the retired Python prototype
> (`podwash.py`, initial commit `688e9a6` — view with `git show 688e9a6:podwash.py`).
> The prototype code is deleted; **this document is the normative spec** the Swift
> port (Slice 02) implements. Referenced from PRD §3 (profanity handling) and
> PRD §5 (word and category selection).
>
> The golden example at the bottom was **computed by hand from these rules**, not
> generated by any implementation — it gives test goldens independent provenance.

## 1. Constants (normative)

| Constant | Value | Meaning |
|----------|-------|---------|
| `START_PADDING_SECONDS` | **0.080** | Subtracted from each matched word's start (clamped at 0.0) |
| `END_PADDING_SECONDS` | **0.120** | Added to each matched word's end |
| `MIN_CENSOR_SECONDS` | **0.180** | Minimum duration of a single censor interval |

Non-normative prototype constants for the (deferred) beep overlay, recorded for the
future overlay slice: `BEEP_FREQUENCY_HZ = 1000.0`, `BEEP_VOLUME = 0.35`,
`BEEP_FADE_SECONDS = 0.005`. iOS playback-time overlay (AVAudioEngine) will define
its own ramps; these are starting values only.

## 2. Input: word-level transcript

The matcher consumes the shared transcript schema (see ADR-000):

```json
[ { "word": "Shit!", "start": 3.00, "end": 3.20 }, ... ]
```

- `word` — raw token as produced by ASR (may include punctuation and casing).
- `start` / `end` — seconds from episode start (Double).
- Words missing `start` or `end` are dropped before matching.

## 3. `normalize_word` (normative)

Python original:

```python
def normalize_word(word: str) -> str:
    normalized = word.lower().strip()
    return re.sub(r"^[^a-z0-9']+|[^a-z0-9']+$", "", normalized)
```

Behavior, precisely:

1. Lowercase the token, then trim ASCII whitespace.
2. Strip **leading and trailing** runs of any character that is *not* a lowercase
   ASCII letter (`a-z`), digit (`0-9`), or apostrophe (`'`).
3. **Interior** characters are untouched — `"f*ck"` normalizes to `"f*ck"`
   (asterisk kept), `"shit's"` normalizes to `"shit's"` (apostrophe kept, and it
   does **not** equal `"shit"`).

Examples: `"Shit!"` → `"shit"`; `"  FREAKING,"` → `"freaking"`; `"'ship'"` →
`"ship"`; `"$#!%"` → `""` (empty string never matches).

## 4. Matching semantics (normative)

- Matching is **exact set membership**: `normalized_word ∈ target_set`.
- **No stemming, no substring matching, no fuzzy matching.** Inflections are
  handled by *enumerating* them in the word list (e.g. both `"ship"` and
  `"shipped"` are listed). `"shipment"` does NOT match a list containing
  `"ship"` — this is the substring-false-positive guarantee.
- Target lists are normalized with the same `normalize_word` before use.

## 5. Padding and the midpoint-expansion rule (normative)

For each matched word:

```
padded_start = max(0.0, word.start − START_PADDING_SECONDS)
padded_end   = word.end + END_PADDING_SECONDS

if (padded_end − padded_start) < MIN_CENSOR_SECONDS:
    midpoint      = (padded_start + padded_end) / 2
    half_duration = MIN_CENSOR_SECONDS / 2
    padded_start  = max(0.0, midpoint − half_duration)
    padded_end    = midpoint + half_duration
```

**When does the expansion branch fire?** Padding adds 0.080 + 0.120 = 0.200 s,
which already exceeds `MIN_CENSOR_SECONDS` (0.180). So with the default constants
the branch can only fire when the start clamp at 0.0 truncated the interval —
i.e. for words that *end* before 0.060 s. The rule must still be implemented
exactly as written (constants are user-tunable in principle).

**Known quirk (implement as-is):** inside the expansion branch the *new*
`padded_start` is clamped at 0.0 again, but `padded_end` is **not** re-extended
to compensate, so an interval hugging t=0 may still end up shorter than
`MIN_CENSOR_SECONDS`. Example: word at `[0.00, 0.05]` → padded `[0.00, 0.17]`
(0.17 < 0.18) → midpoint 0.085, half 0.09 → start `max(0, −0.005) = 0.0`, end
`0.175` → final `[0.000, 0.175]`, still 0.175 s. The Swift port replicates this
behavior; the golden fixtures encode it.

## 6. Sort-and-merge (normative)

```
intervals = padded intervals of all matches, sorted ascending by start
merged    = []
for each interval in intervals:
    if merged is nonempty and interval.start <= merged.last.end:
        merged.last.end = max(merged.last.end, interval.end)   # absorb
    else:
        merged.append(interval)
```

- Merge condition is `<=`: intervals that merely **touch** (start == previous
  end) merge into one.
- Merged end is `max(previous.end, interval.end)` — a fully-contained interval
  does not shorten its container.
- Output is an ordered, disjoint list of `{start, end}` intervals. Actions
  (mute/skip) attach per interval downstream; the merge itself is action-agnostic
  in the prototype (the Swift `IntervalBuilder` merges within one action list).

## 7. Seeded word lists (from the prototype)

These seed the default category profiles (PRD §5). Lists are user-configurable at
runtime; these are the shipped starting points and the values used by fixtures.

```
harmless  (test profile): freak, freaking, ship, shipped
profanity (seed profile): fuck, fucked, fucker, fuckers, fucking, fucks,
                          shit, shits, shitty, bullshit
```

The prototype's default `TARGET_WORDS` equals the `harmless` profile — useful for
building clean-language test fixtures.

## 8. Hand-computed golden example (independent provenance)

Computed by hand from Sections 3–6 on 2026-07-08. Any implementation that
disagrees with this walkthrough is wrong — do **not** regenerate these numbers
from code under test.

**Target set** (example-local): `{ "shit", "damn" }`

**Input transcript:**

```json
[
  { "word": "Well,",     "start": 0.50, "end": 0.90 },
  { "word": "shit,",     "start": 1.00, "end": 1.30 },
  { "word": "damn",      "start": 1.50, "end": 1.75 },
  { "word": "shipment",  "start": 2.00, "end": 2.60 },
  { "word": "Shit!",     "start": 3.00, "end": 3.20 }
]
```

**Step 1 — normalize + match:**

| Token | Normalized | In target set? |
|-------|-----------|----------------|
| `Well,` | `well` | no |
| `shit,` | `shit` | **yes** |
| `damn` | `damn` | **yes** |
| `shipment` | `shipment` | no (no substring match on `shit`/`ship`) |
| `Shit!` | `shit` | **yes** |

**Step 2 — pad each match** (start − 0.080, end + 0.120; all durations ≥ 0.180 so
no midpoint expansion):

| Word | Raw `[start, end]` | Padded `[start, end]` | Duration |
|------|--------------------|-----------------------|----------|
| `shit,` | [1.00, 1.30] | [0.92, 1.42] | 0.50 |
| `damn` | [1.50, 1.75] | [1.42, 1.87] | 0.45 |
| `Shit!` | [3.00, 3.20] | [2.92, 3.32] | 0.40 |

**Step 3 — sort (already sorted) and merge:**

- `[0.92, 1.42]` opens the list.
- `[1.42, 1.87]`: start 1.42 **<=** previous end 1.42 → merge (touch counts) →
  `[0.92, max(1.42, 1.87)] = [0.92, 1.87]`.
- `[2.92, 3.32]`: start 2.92 > 1.87 → new interval.

**Expected merged output (golden):**

```json
[
  { "start": 0.92, "end": 1.87 },
  { "start": 2.92, "end": 3.32 }
]
```

**Supplementary hand-computed case — clamp + midpoint expansion:** word `"shit"`
at `[0.00, 0.05]`, same target set → padded `[max(0, −0.08), 0.17] = [0.00, 0.17]`
→ duration 0.17 < 0.18 → midpoint 0.085, half 0.09 → `[max(0, −0.005), 0.175]` →
**final `[0.000, 0.175]`** (see the Section 5 quirk).

Slice 02 fixture files encode both cases; their expected-output JSON must match
these numbers exactly (tolerance for Double comparison: ±0.0005 s).
